Hello World 출력에도 100MB 의 JRE 가 필요한 자바는 MSA 의 추세와 잘 맞지 않는다.
이를 위해 JDK 에는 클래스 데이터 공유 (Application Class Data Sharing, AppCDS) 와 노옵(no-op) 가비지 컬렉터인 엡실론 등의 기술이 포함되었다.

AppCDS: 로딩한 클래스 정보를 캐시해두어 다음번 구동 시간을 줄이는 기술, 초기에는 자바 표준 라이브러리만 지원했지만 JDK10 버전부터 사용자 프로그램의 코드까지 지원
- 그럼 클래스 정보가 바뀔 때는 새로 컴파일 하는 것?
- CI 단계에서 일부 캐싱을 통해 빌드 시간을 최적화 하는 것과 비슷한 기술?

엡실론: 메모리를 할당만 해주고 회수는 하지 않는 컬렉터로, 간단한 작업을 빠르게 처리한 후 즉시 종료하는 애플리케이션에 적합

AOT: 애플리케이션을 실행하기 전에 네이티브 코드로 컴파일 해두는 AOT(ahead of time) 컴파일
JVM 은 애플리케이션을 우선 실행 한후 JIT 컴파일러를 써서 빈번하게 실행되는 로직을 네이티브 코드로 바꿔 실행했다. 하지만 미리 컴파일 해두면 처음부터 네이티브 코드를 실행할 수 있다. -> 시작시간이 느린 자바의 단점을 보완 -> 하드웨어 및 운영체제별로 컴파일 작업되어야 하기 때문에 WORA 의 원칙이 지켜지지 않음
