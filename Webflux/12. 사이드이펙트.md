`map`, `filter` 와 같은 연산자는 데이터나 흐름 자체를 변경한다.
반면 doOn 계열 연산자는 스트림의 흐름이나 데이터에 영향을 주지 않고 특정 시점/신호에 데이터를 관찰만 하는 부수 효과(side Effect)용 연산자다.
로깅, 메트릭 기록, 디버깅에 주로 사용된다.

# doOnNext
각 데이터가 발행될 때 마다 실행된다

```
Flux.just("apple", "banana", "cherry")
  .doOnNext(data -> System.out.println("처리 중 : " + data))
  .map(String::toUpperCase)
  .doOnNext(data -> System.out.println("처리 중 2 : " + data))
  .subscribe(data -> System.out.println("최종 결과 : " + data));
// 처리 중 : apple
// 처리 중 2 : APPLE
// 최종 결과 : APPLE
// 처리 중 : banana
// 처리 중 2 : BANANA
// 최종 결과 : BANANA
// 처리 중 : cherry ...
```

리액티브 스트림은 원소 하나가 파이프라인 전체 (doOnNext -> map -> doOnNext -> subscribe) 를 다 통과한 뒤에야 다음 원소가 시작된다.
apple 이 처리되고 banana 가 시작되는 것으로, 모든 원소 doOnext 먼저 -> 전체 map 같은 배치 처리가 아니다.

# doOnError
에러 신호가 발생했을 때 트르거된다.
단, 에러를 처리(복구) 하지 않고 관찰만 한다 -> 로그를 찍어도 에러는 그대로 구독자의 에러 콜백까지 전파된다.

```
Flux.just(1, 2, 0, 4)
  .map(n -> 10 / n)
  .doOnError(error -> System.out.println("에러 발생 : " + error.getMessage()))
  .subscribe(
	  data -> System.out.println("데이터 " + data),
	  error -> System.out.println("구독자에서 에러 발생 : " + error.getMessage()));
// 데이터 10, 데이터 5, 에러 발생 : / by zero, 구독자에서 에러 발생 : / by zero
```

실제로 에러를 복구/대체 하려면 doOnError 가 아니라 앞선 onErrorReturn/onErrorResume 같은 에러 처리 연산자를 써야한다.
그래서 실무에서는 보통 로깅용 doOnError + 복구용 onErrorReturn/onErrorResume 을 함께 체이닝한다.

```
Flux.just(1, 2, 0, 4)
  .map(n -> 10 / n)
  .doOnError(error -> System.out.println("에러 발생 : " + error.getMessage())) // 관찰(로깅)
  .onErrorReturn(-1) // 실제 복구
  .subscribe(
	  data -> System.out.println("데이터 " + data),
	  error -> System.out.println("구독자에서 에러 발생 : " + error.getMessage()),
	  () -> System.out.println("완료"));
// 데이터 10, 데이터 5, 에러 발생 : / by zero, 데이터 -1, 완료
```

# 기타 doOn 계열 연산자
| 연산자        | 트리거 시점                                     |
| ------------- | ----------------------------------------------- |
| doOnSubscribe | 구독이 시작될때                                 |
| doOnCancel    | 구독이 취소될 때                                |
| doOnEach      | 데이터/에러/완료 등 모든 신호에 대해            |
| doFirst       | 구독 직전                                       |
| doFinally     | 스트림이 어떤 이유로든(완료/에러/취소) 종료될때 |

트리거 되는 시점만 다를 뿐 전부 관찰만하고 흐름엔 영향을 주지 않는다

# 주의사항
- doOn 계열은 스트림을 직접 제어하지 않지만, 콜백 내부에서 예외가 발생하면 그 예외는 그대로 스트림을 에러로 종료시킨다. 
- doOn 콜백 안에서 예외가 안터지게 주의해야한다.
