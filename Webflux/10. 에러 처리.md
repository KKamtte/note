에러가 발생하였을 때 `subscribe()` 의 에러 콜백이 구현되어 있지 않으면, Reactor가 그 에러를 그대로 throw 해버린다.
그래서 에러 콜백 구현은 필수적이며, 콜백보다 앞단에서 에러를 후속 처리하는 패턴이 있다.

# onErrorReturn
에러가 발생했을 때 지정된 단일 대체값을 반환한다
- 대체값을 발행한 뒤 스트림은 정상종료 (onComplete 호출) 된다. onError 콜백은 실행되지 않는다.
- 단, 에러 발생 이후 나머지 원소는 처리하지 않는다. 에러 시점에 스트림 자체가 종료되고 그 자리에 대체값 하나가 끼워진다.

```
Flux.just(1, 2, 0, 4, 5)
  .map(n -> 10 / n)
  .onErrorReturn(-1)
  .subscribe(
	  data -> System.out.println("Received: " + data),
	  error -> System.err.println("Error: " + error),
	  () -> System.out.println("Completed"));
// 10, 5, -1, Completed  (4, 5는 처리 안 됨)
```

특정 예외 타입에 대해서만 선택적으로 적용할 수도 있다. `(onErrorReturnClass<E>, value))`
명시한 타입이 아닌 예외가 발생하면 적용되지 않고 에러가 그대로 subscriber 까지 전달된다.

```
.onErrorReturn(ArithmeticException.class, -1)   
// 발생 예외와 일치 → 10, 5, -1, Completed
.onErrorReturn(NullPointerException.class, -1)  
// 발생 예외와 불일치 → 10, 5, Error: ...
```

# onErrorResume
에러가 발생했을 때 다른 `Publisher`로 전환한다 (단일 값이 아닌 대체 스트림 제공)

```
Flux.just(1, 2, 0, 4, 5)
  .map(i -> 10 / i)
  .onErrorResume(e -> {
	  System.out.println("에러 발생: " + e.getMessage());
	  return Flux.just(100, 200, 300); // 대체 스트림
  })
  .subscribe(
	  data -> System.out.println("데이터: " + data),
	  error -> System.out.println("구독 중 에러: " + error.getMessage()),
	  () -> System.out.println("구독 완료"));
// 데이터: 10, 데이터: 5, (에러 발생: ...), 데이터: 100, 200, 300, 구독 완료
```

`OnErrorReturn` 과 마찬가지로 특정 예외 타입에 한정해서 적용하는 오버로드 존재

# onErrorMap
에러를 다른 에러로 변환한다.
대체값이나 대체 스트림을 제공하는게 아니라 에러 타입/메시지만 바꾸는것으로 스트림은 에러로 종료된다.

```
Flux.just(1, 2, 0, 4, 5)
  .map(n -> 10 / n)
  .onErrorMap(ArithmeticException.class,
	  e -> new IllegalArgumentException("0으로 나눌 수 없습니다.", e))
  .subscribe(
	  System.out::println,
	  error -> {
		  System.out.println("에러 발생: " + error.getMessage());
		  System.out.println("에러 타입: " + error.getClass().getSimpleName());
	  });
// 10, 5, 에러 발생: 0으로 나눌 수 없습니다., 에러 타입: IllegalArgumentException
```

하위 계층의  기술적 예외(`SQLException` 등)를 상위 계층의 비즈니스 예외로 변환할 때 사용된다.
기능적 차이가 아닌 유지보수/디버깅/로깅 관점에서 에러 메시지를 더 이해하기 쉽게 만드는것이 목적이다.

# 연산자 비교
| 연산자        | 대체 방식                  | 스트림 종료 신호        | 주 사용 목적                    |
| ------------- | -------------------------- | ----------------------- | ------------------------------- |
| onErrorReturn | 단일 대체값                | onComplete              | 간단한 fallback 값              |
| onErrorResume | 대체 Publisher (Mono/Flux) | 대체 스트림의 종료 신호 | 캐시 조회, 재시도 등 복구 로직  |
| onErrorMap    | 에러 -> 다른 에러          | onError (변환된 에러)   | 기술 예외 -> 비즈니스 예외 변환 |

# 체이닝 순서
에러 처리 연산자는 선언 순서대로 적용된다.
- `onErrorResume` 이 먼저 에러를 잡고 정상적으로 대체 스트림을 변환하면 -> 그 뒤의 `onErrorMap` 은 실행되지 않는다.
- `onErrorResume` 의 대체 로직 안에서 다시 `Mono.error(...)/Flux.error(...)` 를 반환하면 -> 그 에러가 다음 `onErrorMap` 으로 전달 된다.

# 조합 예시 - 캐시 fallback + 에러 변환

```
Map<String, String> cache = Map.of("user-1", "홍길동(캐시)");

// case 1: 캐시에 있음 → 정상 복구
findUserFromDb("user-1")
  .onErrorResume(error -> {
	  System.out.println("DB 에러, 캐시 조회: " + error.getMessage());
	  String cached = cache.get("user-1");
	  return cached != null ? Mono.just(cached) : Mono.error(error);
  })
  .subscribe(user -> System.out.println("결과: " + user));
// DB 에러, 캐시 조회: DB 연결 실패
// 결과: 홍길동(캐시)

// case 2: 캐시에도 없음 → 에러 재전달 → onErrorMap이 비즈니스 에러로 변환
findUserFromDb("user-2")
  .onErrorResume(error -> {
	  System.out.println("DB 에러, 캐시 조회: " + error.getMessage());
	  String cached = cache.get("user-2"); // 캐시에 없음 → null
	  return cached != null ? Mono.just(cached) : Mono.error(error);
  })
  .onErrorMap(error -> new RuntimeException("사용자 조회 실패: " + error.getMessage()))
  .subscribe(
	  user -> System.out.println("결과: " + user),
	  error -> System.out.println("최종 에러: " + error.getMessage()));
// DB 에러, 캐시 조회: DB 연결 실패
// 최종 에러: 사용자 조회 실패: DB 연결 실패

static Mono<String> findUserFromDb(String userId) {
  return Mono.error(new RuntimeException("DB 연결 실패"));
}
```

- user-1: 캐시에서 발견 -> `onErrorResume` 이 성공적으로 복구 -> 뒤에 `onErrorMap` 이 있어도 실행이 안됨
- user-2: 캐시에도 없음 -> `onErrorResume` 이 에러를 다시 던짐 -> 그 에러가 `onErrorMap` 까지 전달되어 비즈니스 메시지로 변환된 후 최종 에러 콜백 호출
