# filter
조건이 true 인 데이터만 통과
체이닝 시 AND 조건으로 동작
조건별로 분리하는 체이닝 방식이 복합 조건보다 명시적이어서 권장된다

```
Flux.just(...)
  .filter(p -> p.inStock())
  .filter(p -> p.price() <= 100000)
  .subscribe(...);
```

# filterWhen
조건 함수가 `Mono<Boolean>` 을 반환하는 비동기 필터링
DB 조회, 외부 API 호출처럼 조건 확인 자체가 비동기 작업일 경우 사용된다

```
Flux.just("user-1", "user-2", "user-3")
  .filterWhen(userId -> hasPermission(userId)) // Mono<Boolean> 반환
  .subscribe(...);
```

|                | filter         | filerWhen       |
| -------------- | -------------- | --------------- |
| 조건 반환 타입 | boolean        | Mono\<Boolean\> |
| 실행 방식      | 동기           | 비동기          |
| 사용 상황      | 단순 조건 검사 | DB 조회, 외부 API 호출                |

# distinct
중복 데이터 제거
내부적으로 지나간 값을 메모리에 기록하므로 대용량 스트림에서 OOM 주의해야한다

```
// 기본: equals/hashCode 기반
Flux.just(1, 2, 3, 2, 1).distinct();

// 키 기반: 특정 필드 기준으로 중복 제거. 최초 등장한 값이 유지됨
Flux.just(...).distinct(user -> user.city());
```

# distinctUntilChanged
전체 중복이 아닌 연속된 중복만 제거
전체 이력을 메모리에 보관하지 않아 대용량 스트림에 적합하다

```
Flux.just(1, 1, 2, 2, 1, 3)
  .distinctUntilChanged()
  .subscribe(...); // 1, 2, 1, 3
```

# take / skip
```
Flux.range(1, 10).take(3);  // 앞에서 3개만 통과, 이후 스트림 취소
Flux.range(1, 10).skip(3);  // 앞에서 3개를 건너뜀
```

# elementAt
인덱스로부터 특정 위치 데이터를 Mono로 반환
범위 초과 시 `IndexOutOfBoundException` 발생 -> 두 번째 인자로 기본값을 처리
데이터가 흐르는 리액티브 패러다임과 맞지는 않는다

```
Flux.just("사과", "바나나", "체리").elementAt(2);          // 체리
Flux.just("사과", "바나나").elementAt(5, "없음");          // 없음
```
