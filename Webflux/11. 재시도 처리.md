# retry
에러가 발생하면 원본 Publisher를 다시 구독하는 가장 단순한 재시도 정책
인자로 최대 재시도 횟수만 지정한다.
재구독은 원본 로직을 처음부터 다시 실행하는 것이므로, 매 구독 시점마다 새로 평가되는 `Mono.defer()` 와 함께 사용해야 의미가 있다. `Mono.just()` 로 이미 계산된 값을 감싸면 재구독해도 같은 결과만 반복된다.

```
AtomicInteger callCount = new AtomicInteger();

Mono.defer(() -> {
  int count = callCount.incrementAndGet();
  System.out.println("호출 # " + count);

  if (count < 3) {
	  return Mono.error(new RuntimeException("일시적인 에러"));
  }

  return Mono.just("성공");
})
  .retry(2) // 최대 2번 재시도 (최초 1번 + 재시도 2번 = 총 3번 시도)
  .subscribe(
	  data -> System.out.println("데이터: " + data),
	  error -> System.out.println("에러: " + error.getMessage()));
// 호출 # 1, 호출 # 2, 호출 # 3, 데이터: 성공
```

재시도 횟수를 초과하면 마지막 에러가 그대로 subscriber에 전달된다.
retry는 일시적인 장애를 막아주는 방패일 뿐, 횟수를 넘기면 더이상 기회를 주지 않고 실패를 그대로 흘려보낸다.
인자를 생략하면 (retry()) 무한 재시도하므로 위험하다 -> 반드시 횟수를 제한해야 한다.

### retry 한계
- 에러 발생 즉시 재시도 한다 -> 일시적 장애(서버 과부하 등)가 해소될 시간을 주지 못하고, 오히려 짧은 간격으로 계속 요청을 보내 과부하를 유발할 수 있다.
- 모든 에러를 동일하게 재시도한다 -> 400 Bad Request 처럼 재시도해도 절대 성공할 수 없는 에러까지 무의미 하게 반복하게 된다.

# retryWhen
retry 전략 객체를 받아 재시도 정책(간격, 횟수, 조건 등)을 세밀하게 구성할 수 있는 메서드

```
AtomicInteger callCount = new AtomicInteger(0);
long start = System.currentTimeMillis();

Mono.defer(() -> {
  int count = callCount.incrementAndGet();
  long elapsed = System.currentTimeMillis() - start;
  System.out.println("callCount: " + count + ", elapsed: " + elapsed + "ms");

  if (count < 4) {
	  return Mono.error(new RuntimeException("일시적 오류 발생"));
  }
  return Mono.just("성공");
})
  .retryWhen(Retry.fixedDelay(5, Duration.ofMillis(500))) // 최대 5번, 매 시도 사이 500ms 대기
  .subscribe(
	  result -> System.out.println("결과: " + result),
	  error -> System.err.println("최종 오류: " + error.getMessage()));

Thread.sleep(3000);
```

`Retry.fixedDelay(5, Duration.ofMillis(500))` 최대 5번 까지, 매 재시도 사이에 500ms 간격을 강제로 둔다. 일시적 장애가 해소될 여유 시간을 벌어준다.

retryWhen 의 지연(delay)은 별도의 스레드에서 처리된다.
main 스레드가 먼저 종료되면 재시도 전에 프로그램이 끝나버린다.

### filter - 특정 예외만 재시도
모든 에러를 재시도하는 건 비효율적이다. 예를 들어 400 Bad Request 처럼 재시도해도 성공할 수 없는 에러는 재시도 대상에서 제외해야 한다.
retry 전략에 `.filter()` 를 추가하면 특정 예외 타입에 대해서만 재시도 하도록 제한할 수 있다

```
.retryWhen(Retry.fixedDelay(3, Duration.ofMillis(500))
  .filter(throwable -> throwable instanceof TransientException))
// TransientException 이 아닌 에러는 재시도 없이 즉시 onError로 전파됨
```

### Exponential Backoff (Retry.backoff)
고정 간격(fixedDelay)으로 재시도 하면, 장애 상황에서 여러 클라이언트가 똑같은 간격으로 동시에 재시도를 몰아서 보내 서버 부하를 다시 유발할 수 있다 (thundering herd 문제)
재시도 간격을 점점 늘리는 Retry.backoff() 가 더 많이 사용된다.

```
.retryWhen(Retry.backoff(5, Duration.ofMillis(100))
  .maxBackoff(Duration.ofSeconds(2))
  .jitter(0.5)) // 동시 재시도가 겹치지 않도록 무작위성 추가
```

- 첫 재시도는 약 100ms 후, 이후 200ms, 400ms 식으로 지수적으로 늘어나다가 최대 2초로 제한된다.
- jitter로 지연시간에 무작위성을 섞어, 동시에 실패한 여러 클라이언트가 똑같은 시점에 재시도를 몰아서 보내는 것을 방지한다.

# retry + onErrorResume 조합
재시도를 모두 시도했는데도 실패하면 `onErrorResume` 으로 기본값/캐시값을 반환해 사용자 경험을 지키는 패턴

```
Mono.defer(() -> {
	  System.out.println("외부 API 호출...");
	  return Mono.<String>error(new RuntimeException("서비스 불가"));
  })
  .retryWhen(Retry.fixedDelay(2, Duration.ofMillis(200)))
  .onErrorResume(error -> {
	  System.out.println("모든 재시도 실패, 캐시 값 반환");
	  return Mono.just("캐시된 기본 응답");
  })
  .subscribe(data -> System.out.println("결과: " + data));

Thread.sleep(1000);
```

총 3번 시도 (최초1번 + 재시도2번) 후에도 실패하면 `RetryExhaustedExpcetion` 이 발생하고, 이를 `onErrorResume`이 잡아 캐시값으로 대체한다.

# retry vs retryWhen
| 구분          | retry                   | retryWhen                                                |
| ------------- | ----------------------- | -------------------------------------------------------- |
| 설정          | 최대 횟수만 지정        | Retry 전략 객체로 세밀하게 구성                          |
| 재시도 간격   | 없음 (즉시 재시도)      | 고정 (fixedDelay) / 지수 (backoff) 등 다양하게 설정 가능 |
| 조건별 필터링 | 불가                    | .filter() 로 특정 예외만 재시도 가능                     |
| 사용 시점     | 단순 재시도가 충분할 때 | 장애 회복 시간, 예외 종류 등 세밀한 제어가 필요할 때     |
