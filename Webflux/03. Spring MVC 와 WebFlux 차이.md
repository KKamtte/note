
# Spring MVC - Thread-per-request 모델
Servlet API 기반으로, 요청 하나당 Thread 하나를 할당한다.
```
요청 → Tomcat Thread Pool 에서 Thread 꺼냄 → Filter  → DispatcherServlet  → Controller  → Service  → Repository  → DB 응답이 올때까지 Thread Blocking  → 응답 완료 후 Thread 반환
```
이로 인해 처리량에 한계가 발생한다.
```
┌────────────────────────────────────────┬──────────────┐
│                  상황                   │    처리량      │
├────────────────────────────────────────┼──────────────┤
│ Thread 200개, 요청당 100ms               │ 초당 2,000건   │
├────────────────────────────────────────┼──────────────┤
│ Thread 200개, 요청당 1,000ms (DB 느림)    │ 초당 200건     │
└────────────────────────────────────────┴──────────────┘
```
Thread를 늘리면 스택 메모리 소비와 컨텍스트 스위칭 비용이 커져 한계가 있다.

# Spring WebFlux - Event Loop 모델
소수의 이벤트 루프 Thread (만약 쿼드코어 CPU 라면 4개)가 모든 요청을 처리한다.
```
요청 → 이벤트 루프 Thread가 받아서 처리 시작 → DB 쿼리 전송 후 `응답이 오면 콜백 실행` 등록 → Thread 는 즉시 다른 요청 처리하러 이동 → DB 응답 도착  → 이벤트 루프가 감지 → 콜백 실행 → 응답 반환
```
Thread 가 I/O 대기 없이 항상 일하기 때문에, 적은 Thread 로 수만 개의 동시 연결 처리가 가능하다.
내장서버는 Tomcat 이 아닌 Non-Blocking I/O 를 위해 설계된 Netty 를 사용한다.

# 코드를 통한 비교
```
@RestController
@RequestMapping("/users")
public class UserController {

      @GetMapping
      public List<User> getUsers() {
          return userService.findAll();
      }

      @GetMapping("/{id}")
      public User getUser(@PathVariable Long id) {
          return userService.findById(id);
      }

      @PostMapping
      public User createUser(@RequestBody User user) {
          return userService.save(user);
      }
}
```
반환 타입이 `User` 와 `List<User>` 와 같은 특정 DTO 모델을 사용한다.
메서드가 리턴할 때 이미 DB 에서 데이터를 가져와서 리턴할때 값이 만들어져야하며, 반환 전까지 Thread 가 점유된다.

```
@RestController
@RequestMapping("/users")
public class UserController {

      @GetMapping
      public Flux<User> getUsers() {
          return userService.findAll();
      }

      @GetMapping("/{id}")
      public Mono<User> getUser(@PathVariable Long id) {
          return userService.findById(id);
      }

      @PostMapping
      public Mono<User> createUser(@RequestBody User user) {
          return userService.save(user);
      }
}
```
반환 타입이 `Mono` 와 `Flux` 이며, 이 둘은 나중에 값이 도착할 것이라고 약속하는 것이다.
반환하는 시점에 DB 조회가 아직 실행되지 않았을 수 있다. 실제 실행은 WebFlux가 구독할 때 이루어진다.

# 사용의 차이
### MVC 가 적합한 경우
JDBC, JPA 등 블로킹 방식의 DB 접근 사용시
CPU 집약적이고 I/O 대기가 짧은 작업
이미 잘 동작하고 처리량이 충분한 서비스
### WebFlux 가 적합한 경우
동시 연결이 많고 하드웨어 리소스를 아껴야 할때
MSA G/W 처럼 다수 요청을 다른 서비스로 라우팅할 때
외부 API 를 여러개 동시에 호출하고 결과를 조합할 때
I/O 대기 시간이 긴 작업 (느린 외부 API, 대량 DB 쿼리)
Server-Sent Events, WebSocket, 실시간 스트리밍 등 장시간 연결 유지가 필요한 경우

# WebFlux 에서 블로킹 코드 금지
MVC 는 Thread 하나가 블로킹되면 요청 한개가 영향을 받지만, WebFlux 는 이벤트 루프 Thread 하나가 블로킹 되면 해당 Thread 에 할당된 수천 개 요청이 영향을 받는다.

| 금지 항목          | 대체 방법             |
| ------------------ | --------------------- |
| JDBC,JPA           | R2DBC                 |
| RestTemplate       | WebClient             |
| Thread.sleep()     | Mono.delay()          |
| Synchronized Block | Reactor 동시성 제어   |
| 블로킬 파일 I/O    | Non-Blocking 파일 I/O |
