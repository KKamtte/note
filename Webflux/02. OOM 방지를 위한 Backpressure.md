수십만의 데이터가 쏟아지는 상황에서 서버가 초당 1000건만 처리할 수 있다면, 처리되지 못한 데이터는 내부 버퍼에 쌓아두게 된다.
버퍼가 계속 커지면서 메모리 사용량이 증가하고, 결국 Out of Memory 가 발생하여 애플리케이션이 다운될 수 있다.
외부 API 에서 데이터를 받아와 가공하는 파이프라인, 로그를 수집하고 처리하는 시스템, EDA 등에서 생산하는 과정과 소비하는 속도의 불일치는 항상 존재한다. (Kafka - Consumer Lag 현상)

데이터를 소비하는 쪽에서 생산하는 쪽에게 처리할 수 있는 만큼 요청하는 방식을 Backpressure 라고 한다.

```
  Subscriber          Publisher          Subscription
      |                   |                   |
      |---subscribe()---->|                   |
      |                   |---new------------>|
      |<--onSubscribe()---|<------------------|
      |                   |                   |
      |---request(3)------------------------->|
      |                   |<--trigger---------|
      |<--onNext(item1)---|                   |
      |<--onNext(item2)---|                   |
      |<--onNext(item3)---|                   |
      |                   |                   |
      |---request(2)------------------------->|
      |                   |<--trigger---------|
      |<--onNext(item4)---|                   |
      |<--onComplete()----|                   | ← 남은 데이터 없음, 완료
```

# 데이터 흐름 패턴

### Push 방식 - Observer 패턴
이벤트가 발생하면 옵저버에게 일방적으로 밀어넣는 방식

### Pull 방식 - Iterator
데이터를 호출해서 가져오는 방식
데이터가 준비될 때까지 일부 블로킹이 되어야한다

### Hybrid 방식 - Reactive Streams
Subscriber 가 request 로 데이터를 요청한다 (pull)
Publisher 가 데이터를 준비되는대로 보내준다 (push)ㄱ
Subscriber 가 속도를 제어하지만 데이터 전달은 Non-Blocking 방식으로 이루어진다.

# 데이터 보관 방식
### Buffering - 일반적
아직 요청되지 않는 데이터를 내부 버퍼에 저장해주는 방식
subscriber 가 나중에 request 를 호출하면 버퍼에서 꺼내 전달하는 방식
버퍼 크기에 제한을 두지 않게 되면 OOM 이 발생한다

### Drop
Subscriber 가 아직 Request 하지 않은 상태에서 새로운 데이터가 생산되면 버리는 방식
실시간 데이터 스트림, Apache Flink 와 같은 곳에 사용

### Latest
drop 과 유사한데, 가장 최근에 생산된 데이터 하나만 유지
센서 데이터, 주가 정보, 위치 정보와 같이 현재 값만 봐야 되는 경우 적합

### Error
Subscriber 가 처리하지 못하는 속도로 데이터가 쌓이는 경우 Error 를 발생시켜서 Subscriber 에게 상황을 알린다.
